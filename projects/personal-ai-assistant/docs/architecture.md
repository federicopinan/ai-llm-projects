# Architecture

## Goal

The architecture of this PoC is intentionally minimal. It optimizes for three properties:

1. **Inspectability** — every step is readable in plain text.
2. **Editability** — workflows can be modified without touching code.
3. **Reversibility** — the assistant never takes an action that cannot be undone by the user.

## High-Level Flow

```txt
User objective
      ↓
Workflow selection (markdown file)
      ↓
Prompt template + context fields
      ↓
Context injection (CLI flags, stdin, or .env)
      ↓
Final structured prompt
      ↓
LLM backend (optional)
      ↓
Reviewable output
```

Each stage is described below.

## Stage 1: User Objective

The user states what they want to accomplish. This can be:

- A free-form sentence (`--input "..."`)
- A set of structured fields (`--context key=value ...`)
- A combination of both

The CLI does not interpret this text. It only carries it through to the prompt template.

## Stage 2: Workflow Selection

The user selects a workflow by name:

```bash
python src/assistant.py --workflow daily-planning
```

Each workflow is a markdown file under `workflows/` with the same structure:

- **Purpose** — what the workflow is for
- **Input format** — what context the workflow expects
- **Prompt template** — the prompt the LLM will receive
- **Expected output structure** — the shape of the answer
- **Example** — a worked sample

Workflows are pure data. They are loaded by `workflow_loader.py` and substituted at runtime.

## Stage 3: Prompt Template + Context Fields

The workflow markdown file declares placeholders that the assistant fills in at run time. Common placeholders include:

- `{{user_input}}` — the raw user objective
- `{{available_time}}` — time budget for the task
- `{{priorities}}` — explicit list of priorities
- `{{energy_level}}` — high, medium, low
- `{{constraints}}` — any hard constraints (deadlines, dependencies)

The set of placeholders is open-ended. Each workflow declares its own.

## Stage 4: Context Injection

Context is injected in three ways, in increasing order of specificity:

1. **Free-form input** via `--input` — passed as `{{user_input}}`
2. **Structured fields** via `--context key=value` — each becomes `{{key}}`
3. **Environment defaults** via `.env` — only used for global config (model name, mode)

The CLI warns when a placeholder has no value but the workflow expects one.

## Stage 5: Final Structured Prompt

The substituted prompt is printed to stdout. The default mode is **`prompt-only`**: the assistant stops here.

This is deliberate. The user can:

- Copy the prompt into any LLM chat interface
- Pipe it to a different tool
- Inspect it for clarity before sending it anywhere

## Stage 6: LLM Backend (Optional)

If `ASSISTANT_MODE=ollama`, the assistant posts the prompt to a local Ollama instance and prints the response.

```txt
OLLAMA_BASE_URL=http://localhost:11434
LLM_MODEL=llama3.1
ASSISTANT_MODE=ollama
```

The HTTP call is intentionally simple: a single `POST /api/generate` request. There is no streaming, no tool use, no agent loop. The point is to demonstrate the wiring without coupling the PoC to any specific model provider.

If the assistant cannot reach Ollama, it falls back to printing the prompt and explaining why no LLM was invoked.

## Stage 7: Reviewable Output

Every output — whether generated by an LLM or not — is intended to be reviewed by the user before it is acted on.

The assistant does not:

- Send emails
- Create calendar events
- Modify files
- Call external APIs (other than the optional LLM backend)
- Make decisions on the user's behalf

This is enforced by design. See [`design-decisions.md`](design-decisions.md) for the rationale.

## Module Layout

```txt
src/
├── assistant.py          # CLI entry point
├── workflow_loader.py    # Reads and renders workflow markdown
└── config.py             # Loads .env and exposes runtime config
```

### `assistant.py`

Parses CLI arguments, lists workflows, validates selections, delegates prompt rendering to `workflow_loader.py`, and either prints the prompt or sends it to the configured LLM backend.

### `workflow_loader.py`

Reads the markdown workflow file, extracts the prompt template section, substitutes placeholders, and returns the rendered prompt.

### `config.py`

Loads `.env` using `python-dotenv`. Exposes `OLLAMA_BASE_URL`, `LLM_MODEL`, and `ASSISTANT_MODE` as module-level constants.

## Non-Goals

- This PoC is not a multi-tenant SaaS.
- It is not a memory-augmented agent.
- It is not a calendar or email assistant.
- It is not optimized for any specific model.

It is a **prompt-layer artifact** designed to make the structure of personal AI assistance visible and editable.
